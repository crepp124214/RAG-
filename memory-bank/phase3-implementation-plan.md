# 第三阶段实施计划：多模态 RAG 前后端闭环

> **阶段状态更新（2026-04-06）：** 本计划已立项，当前处于第三阶段启动与实施前准备状态。保留本文件是为了固定第三阶段边界、方案和实施顺序；阶段进展与最新验证结果统一记录在 `progress.md`。

## 目标

在现有文本 RAG 与第二阶段 Tool Calling 基线之上，完成“PDF 图片/图表抽取 -> Qwen-VL 描述 -> 视觉块向量化入库 -> 混合检索 -> 聊天回答引用 -> 前端最小展示”的第三阶段最小闭环。

## 边界

### 本阶段必须实现

- 仅覆盖 PDF 多模态，不处理 DOCX 图片
- 视觉内容以独立视觉块入库，不混入文本块
- 在现有 `parser_service -> chunking_service -> document_ingestion` 主链内扩展多模态能力
- 扩展检索与问答编排，支持视觉块与文本块统一召回
- 扩展聊天引用结构，支持视觉引用
- 在现有前端工作台中展示视觉引用卡片，不新增独立页面

### 本阶段明确不做

- DOCX 图片多模态
- 双表示并存的复杂检索
- 独立多模态资产页
- `python_executor`
- `GraphRAG`
- `Neo4j`
- `Celery`
- `Chroma`

## 关键默认值

- 第三阶段只处理 PDF 内图片、图表和扫描页
- 视觉内容只保留一份正式描述，不生成重复事实
- 单张图片描述失败默认不中断整篇文档，只记录日志并跳过该图片
- 若整篇 PDF 没有文本且所有视觉描述都失败，则文档进入 `FAILED`
- SSE 不新增事件种类，多模态只体现在 `citations` 负载中
- 若后端暂不提供图片预览地址，前端先以文字型视觉引用卡片交付

## 实施顺序

### 步骤 1：扩展多模态入库主链

- 在现有解析链中增加视觉资产提取和描述生成
- 文档状态机新增 `VISUAL_EXTRACTING`
- 保持文档处理仍由 `document_ingestion` 统一编排

### 步骤 2：扩展数据模型与接口契约

- 扩展 `chunks` 以承载视觉块最小元数据
- 必要时新增轻量 `document_assets` 表，仅存视觉资产元数据
- 扩展文档详情和聊天引用结构，支持视觉引用字段

### 步骤 3：扩展检索与问答编排

- 文本块和视觉块统一进入召回池
- 重排后统一返回
- 问答上下文支持视觉描述与页码

### 步骤 4：接入前端最小闭环

- 聊天工作台支持视觉引用卡片
- 文档管理区支持最小多模态可观测信息
- 消息刷新恢复后仍保留视觉引用

### 步骤 5：完成第三阶段验收

- 自动化测试覆盖多模态解析、入库、检索、问答、前端展示
- 完成 PDF 图表场景的人工验收
- 同步更新 `progress.md`、`architecture.md`、`CLAUDE.md`

## 接口与类型变化

- 文档状态新增：`VISUAL_EXTRACTING`
- `chunks` 新增视觉元数据字段，如 `asset_index`、`bbox` 或等价结构
- 文档详情接口可增加 `visual_asset_count`、`has_visual_assets`
- 聊天引用结构新增视觉引用字段，如 `source_type`、`asset_label`、`preview_available`

## 测试要求

- PDF 多模态解析：
  - 有文本无图片
  - 有图片无文本
  - 图文混排
  - 单张图片描述失败
- 入库编排：
  - 文本块与视觉块共同入库
  - 部分视觉失败不影响整篇成功
  - 全量失败转 `FAILED`
- 检索问答：
  - 命中文本块
  - 命中视觉块
  - 混合命中
  - 视觉未命中时保守回答
- API 契约：
  - 文档状态包含 `VISUAL_EXTRACTING`
  - 聊天 `citations` 支持视觉引用
  - 消息持久化与刷新恢复保留视觉引用
- 前端：
  - 视觉引用卡片展示
  - 无缩略图时文字降级
  - 刷新恢复

## 完成标准

- 上传包含图表的 PDF 后，状态能经过 `VISUAL_EXTRACTING` 并最终进入 `READY`
- 用户可以询问“第几页图表表达了什么”，系统能基于视觉块回答并给出引用
- 纯文本问题不发生行为回退
- 视觉描述失败时系统能降级、留痕、不打坏整次聊天
- 前端刷新后视觉引用卡片仍可恢复
