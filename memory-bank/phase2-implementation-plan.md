# 第二阶段实施计划（Tool Calling 最小闭环）

> **阶段状态更新（2026-04-06）：** 本计划已完成实现、自动化验收和人工验收。保留本文件是为了固定第二阶段边界、方案和实施顺序；阶段进展与最新验证结果统一记录在 `progress.md`。

## 目标

在第一阶段稳定底座上，完成第二阶段 Tool Calling 最小闭环，不改前端整体页面结构，打通“后端门控 -> 模型决策 -> 工具执行 -> 结果回灌 -> 同步/SSE 输出 -> 前端展示与恢复”这条最小主链。

## 边界

### 本阶段必须实现

- 后端统一工具定义、注册、门控与编排
- `web_search`
- `document_lookup`
- 聊天同步接口 `tool_calls`
- SSE `tool_call` / `tool_result`
- 前端工具事件消费与工具卡片展示
- 工具结果与引用的消息持久化恢复

### 本阶段明确不做

- `python_executor`
- 多轮复杂工具链
- 多模态工具主链
- `GraphRAG`
- `Neo4j`
- `Celery`
- `Chroma`
- 新的前端页面形态重构

## 关键默认值

- 后端先决定允许工具集合，再交给模型决定是否调用
- 单轮默认最多 1 次工具调用回合
- 工具留痕采用结构化日志，不新增数据库审计表
- 搜索 Provider 采用“真实 Provider + acceptance/fake Provider”双路径
- `document_lookup` 复用现有 repository/service 与 PostgreSQL/pgvector 数据

## 实施顺序

### 步骤 1：建立统一工具层与编排层

- 目标：在不替换第一阶段聊天主链的前提下，为 `qa_service -> chat_service -> /api/chat/*` 增加工具能力
- 完成标准：
  - 工具定义、注册、参数校验、错误码映射可统一管理
  - 工具编排可处理“直接回答 / 请求工具 / 工具失败降级”

### 步骤 2：落地 `web_search`

- 目标：提供可替换 Provider 的联网搜索能力
- 完成标准：
  - 输入支持 `query` 和可选 `top_k`
  - 输出统一为 `title / url / snippet`
  - 具备 acceptance/fake Provider

### 步骤 3：落地 `document_lookup`

- 目标：提供库内状态查询与内容查询能力
- 完成标准：
  - 支持 `document_id` 文档状态
  - 支持 `task_id` 任务状态
  - 支持文档内容查询并返回文档名、片段、页码、匹配说明

### 步骤 4：扩展聊天协议与接口契约

- 目标：让聊天链路可承载工具调用过程
- 完成标准：
  - `POST /api/chat/query` 返回 `tool_calls`
  - `POST /api/chat/stream` 发出 `tool_call`、`tool_result`
  - `message_end` 返回工具调用汇总

### 步骤 5：接入前端工具展示与恢复

- 目标：让前端工作台能展示工具调用过程，并在刷新后恢复
- 完成标准：
  - 前端服务层与状态仓库能消费工具事件
  - 工具卡片能显示成功与失败状态
  - 刷新后可恢复 `citations` 与 `tool_calls`

### 步骤 6：完成第二阶段验收

- 目标：完成自动化验收、API 级人工联调和前端界面人工验收
- 完成标准：
  - 成功路径、失败路径、空结果路径均有验证证据
  - SSE 顺序和消息恢复行为被验证
  - 开发入口、测试入口和环境说明同步更新

## 测试要求

- 工具注册中心测试
- `web_search` 成功与错误映射测试
- `document_lookup` 状态与内容查询测试
- 工具编排成功、失败、未授权工具请求测试
- 聊天同步接口与 SSE 契约测试
- 前端服务层、状态仓库、聊天工作台测试
- API 级人工联调
- 前端真实页面人工验收

- 第二阶段主链路已完成自动化验收与人工验收
