# 基于《RAG-design-document.md》与《tech_stack.md》的第一阶段实施计划（最小可运行产品）

> **给 AI 开发者的说明：** 这是一份只聚焦第一阶段“稳定底座”和“最小可运行产品”的实施计划。必须严格按顺序执行。每一步都必须先完成验证，再进入下一步。严禁擅自扩大范围，严禁提前实现第二阶段及之后的能力。

**目标：** 将当前基于 Streamlit 的 RAG 演示项目，重构为基于 FastAPI、Vue 3、PostgreSQL、pgvector、RQ、Redis 的最小可运行产品，打通“文档上传 -> 异步处理 -> 向量入库 -> 检索问答 -> 会话持久化”的完整链路。

**架构：** 第一阶段采用前后端解耦架构。前端使用 Vue 3 工作台，后端使用 FastAPI 暴露 API，后台使用 RQ + Redis 处理异步文档任务，数据与向量统一落在 PostgreSQL + pgvector。当前阶段不引入 Tool Calling、多模态、GraphRAG。

**技术栈：** Vue 3、Vite、Element Plus、Pinia、FastAPI、Pydantic、Uvicorn、SQLAlchemy、PostgreSQL、pgvector、RQ、Redis、DashScope / Qwen、DashScope Embedding、BGE Reranker、PyMuPDF、Docker Compose

---

## 固定范围

### 本阶段必须实现

- 前后端解耦
- 新项目目录结构
- FastAPI 后端骨架
- Vue 3 前端骨架
- PostgreSQL + pgvector
- RQ + Redis
- 文档上传
- 异步文档解析、分块、向量化、入库
- 文档状态与任务状态查询
- 基础知识库问答
- 会话与消息持久化
- SSE 流式输出

### 本阶段明确不做

- Tool Calling
- `web_search`
- `document_lookup`
- `python_executor`
- 多模态解析
- Qwen-VL 主链路
- Neo4j
- GraphRAG
- WebSocket
- 权限系统
- 多租户
- 复杂后台管理

### 第一阶段固定接口

- `POST /api/documents/upload`
- `GET /api/documents/{document_id}`
- `GET /api/tasks/{task_id}`
- `DELETE /api/documents/{document_id}`
- `POST /api/chat/sessions`
- `GET /api/chat/sessions`
- `GET /api/chat/sessions/{session_id}/messages`
- `POST /api/chat/query`
- `POST /api/chat/stream`

### 第一阶段固定状态

- `UPLOADED`
- `PARSING`
- `CHUNKING`
- `EMBEDDING`
- `READY`
- `FAILED`

- 集成测试用于验证接口、数据库、队列和状态流转

## 默认约束

- 本计划严格基于 `RAG-design-document.md` 与 `tech_stack.md`
- 第一阶段唯一认可的底座是 `FastAPI + Vue 3 + PostgreSQL + pgvector + RQ + Redis`
- 旧 `core/` 只作为迁移参考，不再继续扩展
- 第二阶段及以后能力必须另行规划
- 实施时不得擅自加入 Tool Calling、多模态、GraphRAG

---

## 完成标准

当以下条件全部成立时，第一阶段视为完成：

- 新主入口不再依赖 Streamlit
- 文档上传改为异步处理
- 文档可成功入库并进入可检索状态
- 用户可通过前端完成基础知识库问答
- 会话与消息完成持久化
- SSE 流式输出可用
- 成功路径与失败路径都有明确、可测试、可观察的行为
